# 개발 워크플로우 (superpowers + cso + ce-compound)

---

## 0. 핵심 원칙 3가지

1. **추적은 문서로** — 작업 진행 현황은 `docs/plans/*.md`의 `- [ ]` 체크박스로 실시간 확인.
2. **TDD는 자동** — 전역 CLAUDE.md 규칙에 따라 구현 시 항상 테스트 먼저.
   (예외: 프로토타입·설정 파일·생성된 코드)
3. **단순 작업은 건너뛰기** — 기획(brainstorming/writing-plans)은 새 기능·복잡한 작업에만.

---

## 1. 표준 워크플로우 (전체 흐름)

```
[기획]   ── 새 기능·설계 필요할 때만 (단순 작업은 생략)
  1. /brainstorming                 아이디어 → 설계 문서
  2. /writing-plans                 설계 → 구현 계획서(체크박스 .md)

[구현]   ── 계획서를 단계별로 실행하는 핵심 루프
  3. /executing-plans               계획 .md를 단계별 실행 + 체크포인트
        │  (각 단계 구현 시)
        ├─ test-driven-development  ← CLAUDE.md 규칙으로 자동 적용
        │     테스트 먼저 → 최소 구현 → 통과 확인
        └─ systematic-debugging     ← 버그·테스트 실패 시 발동
              근본원인 추적 후 수정
  4. /verification-before-completion  "완료"라 말하기 전 실제 검증

[품질]   ── 병합 전 게이트
  5. /requesting-code-review        기능 정확성·요구사항 충족 점검
  6. /cso                           보안 감사(OWASP/STRIDE/시크릿)

[마무리]
  7. /ce-compound                     해결 과정을 docs/solutions/에 지식화 (같은 커밋/PR에 포함)
  8. /finishing-a-development-branch  merge / PR / 정리 방법 결정
```

---

## 2. 단계별 상세

### 1단계 · brainstorming (기획)
- **언제:** 새 기능/컴포넌트를 만들기 전, 무엇을·왜 만들지 불명확할 때.
- **결과물:** 요구사항이 정리된 설계 문서.
- **생략 조건:** "버튼 색 바꾸기" 같은 단순 작업.

### 2단계 · writing-plans (계획)
- **언제:** 설계가 정해진 뒤, 코드를 건드리기 전.
- **결과물:** `docs/plans/YYYY-MM-DD-기능명.md` — 파일 경로·코드 스니펫·`- [ ]` 단계 포함.
- **여기서부터 "추적 문서"가 됨.** 에디터에서 열어두면 진행 현황이 보임.

### 3단계 · executing-plans (구현)
- **언제:** 계획서가 준비된 후.
- **하는 일:** 계획의 각 단계를 순서대로 실행하며 체크박스를 채움.
- **자동 동반:**
  - `test-driven-development` — 구현마다 테스트 먼저 (자동).
  - `systematic-debugging` — 버그가 나면 추측 금지, 근본원인부터.

### 4단계 · verification-before-completion (검증)
- **언제:** "됐다/통과했다"라고 말하기 직전.
- **하는 일:** 검증 명령(`npm run check-all`, `npm run build` 등)을 실제로 돌려
  출력을 확인한 뒤에만 완료 선언. → 거짓 완료 방지.
- **참조:** 검증 명령은 `~/.claude/rules/testing.md`(전역 기본값) 기준.

### 5단계 · requesting-code-review (코드 리뷰)
- **언제:** 작업 완료·병합 직전.
- **하는 일:** 변경분을 요구사항 대비 점검, 문제를 심각도별로 보고.
- **비개발자에게 특히 중요:** 직접 못 잡는 버그·누락을 잡는 안전망.
- **대안:** 내장 `/code-review` 스킬도 같은 목적으로 사용 가능.

### 6단계 · cso (보안 감사)
- **언제:** 코드 품질 확인 후, 배포·병합 전.
- **하는 일:** OWASP Top 10, STRIDE 위협 모델링, 시크릿 탐지, 의존성 공급망 분석을 15단계로 수행.

### 7단계 · ce-compound (지식화)
- **언제:** 어렵게 해결한 문제가 있을 때(매번 아님).
- **하는 일:** 해결 과정을 문서로 남겨 같은 실수 반복 방지.
- **결과물:** `docs/solutions/`에 생성. **명시적으로 호출할 때만 실행.**
- **순서 이유:** finishing 전에 실행해야 지식 문서가 같은 커밋/PR에 포함되어 코드와 함께 리뷰·머지됨. (단, 머지/정리 과정에서 새로 발견한 이슈가 있다면 finishing 이후 추가로 한 번 더 호출 가능.)

### 8단계 · finishing-a-development-branch (마무리)
- **언제:** 구현 끝 + 테스트 통과 + (필요 시) 지식화 완료 후.
- **하는 일:** 이 작업을 merge할지, PR 만들지, 정리할지 구조적으로 안내.

---

## 3. 스킬 호출 방식

- **자동:** 프롬프트 맥락이 스킬 트리거 조건과 맞으면 자동 발동
  (예: "버그 났어" → systematic-debugging).
- **명시:** `/brainstorming`처럼 직접 지정하면 확실히 작동.
- ⚠️ 자동은 100% 보장은 아니므로, 확실히 원하면 **명시 호출**이 안전.

## 별첨 A. 필수는 아니지만 상황에 따라 쓰는 superpowers 스킬

평소 워크플로우엔 불필요하지만, 특정 상황에서 꺼내 쓰면 좋은 스킬들입니다.

| 스킬 | 언제 (상황) | 목적 | 어떻게 |
|------|------------|------|--------|
| **using-git-worktrees** | 한 기능을 현재 작업공간과 **격리**하고 싶을 때 | 메인 작업 보호 | git worktree로 별도 폴더 분리 후 작업 |
| **receiving-code-review** | 리뷰 **피드백을 받았을 때** | 피드백을 기술적으로 검증 후 반영 | 맹목 수용 금지, 의문점은 따져보고 적용 |
| **dispatching-parallel-agents** | **독립적인 2개 이상** 작업을 동시에 굴릴 때 | 병렬 처리로 속도↑ | 서로 의존 없는 작업을 여러 에이전트에 분배 |
| **subagent-driven-development** | **큰 계획**을 Task별로 쪼개 위임할 때 | 대규모 작업 분산 | Task마다 새 에이전트 + 2단계 리뷰(명세→품질) |
| **writing-skills** | **나만의 새 스킬을 만들** 때 | 반복 작업 자동화 | 스킬 작성 모범사례 따라 SKILL.md 생성 |

> ⚠️ **주의:** dispatching-parallel-agents / subagent-driven-development는
> 비개발자 단독 작업엔 과합니다. 또한 전역 설정의 "에이전트는 요청 시에만 생성"
> 규칙과 맞물리므로, **꼭 필요할 때 명시적으로만** 사용하세요.
